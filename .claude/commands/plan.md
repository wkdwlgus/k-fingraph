# /plan — 새 작업 시작 전 계획 수립

## 0. Sprint 진입 점검 (current.md가 새 sprint로 막 갱신됐다면 먼저)

본 프로젝트에서 **sprint = version 단위** (v0 / v0.5 / v1 / ...). `tasks/current.md`
첫 줄의 sprint 이름이 직전 작업 때와 달라졌다면 sprint 진입 시점이다.

이 경우 다음을 먼저 수행:

1. `docs/decisions/README.md` 인덱스 표를 펼친다
2. "재검토 트리거" 컬럼에서 본 sprint 이름이 등장하는 ADR을 모두 펼쳐 읽는다
3. 트리거가 요구하는 재검토(임계값 측정·universe 확장·정책 supersede 등)를
   본 plan의 작업 단위에 first-class로 포함시킨다 — 별도 bullet으로 노출

같은 sprint 내 다음 작업이라면 0단계 생략하고 1로 진행.

## 1. plan 작성

다음 순서로:

1. `tasks/current.md`를 읽고 다음에 할 작업 확인
2. 해당 작업이 손대야 할 파일/모듈 목록을 사용자에게 제시
3. 의존성: 어떤 함수/스키마/문서를 먼저 손봐야 하는가?
4. 변경할 함수의 시그니처를 미리 적어 사용자에게 보여라
5. 테스트 전략: unit/integration/e2e 중 어디에 무엇을 추가할 것인가
6. **사용자가 "go" 또는 "ok" 라고 명시적으로 승인하기 전에는 코드를 수정하지 말 것**

작성 결과물:
- 변경 대상 파일 리스트
- 새 함수 시그니처
- 테스트 추가 계획

작업 시간 추정은 하지 않는다 — 사용자는 코드 작성을 감독만 하며, 본 어시스턴트의
실행 시간은 일반적인 사람 작업의 시간 단위와 의미가 다르다.

승인 후에야 실제 구현 시작.
