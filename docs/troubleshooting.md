### 2026-05-02: KRX 정보데이터시스템 CSV 다운로드의 EUC-KR 인코딩 함정

**증상**
KRX 정보데이터시스템에서 받은 KOSPI 200 구성종목 CSV(파일명 예:
`data_3227_20260502.csv`)를 일반 텍스트 에디터·Python `open()`(기본 utf-8)로
열면 한글 컬럼명(`종목코드`, `종목명`)과 회사명(`삼성전자`)이 모두 깨진 글자로
표시됨. 파일 내용 자체는 정상.

**원인**
KRX는 다운로드 CSV를 EUC-KR(한국 레거시 인코딩)로 출력한다. 현대 환경(macOS·Linux
기본 utf-8)과 인코딩이 다름. 파일 메타데이터에는 인코딩 정보가 없으므로 자동 감지
실패.

**해결**
`iconv -f EUC-KR -t UTF-8` 또는 Python `open(path, encoding="euc-kr")`로 명시적으로
인코딩 지정해 읽기. 정제 후 UTF-8로 다시 저장하면 이후 모든 도구에서 정상 동작.
원본 파일은 `data/raw/krx/`에 보관(.gitignore 대상), 정제본만 `data/reference/`에
커밋.

**교훈**
- 한국 공공기관 다운로드 파일은 인코딩이 EUC-KR(또는 CP949)인 경우가 흔함. 새
  외부 데이터 소스 추가 시 인코딩을 가장 먼저 확인할 것.
- raw 데이터(`data/raw/`)와 정제 데이터(`data/reference/`)를 분리해 관리하면 인코딩
  변환 같은 정규화 단계가 명시적이 되고 재현 가능해짐.
- 운영 코드는 정제본(UTF-8)만 읽도록 강제 — 원본 인코딩 다양성을 코드 경로에서
  격리.

### 2026-05-03: Neo4j Aura 자격증명 — USERNAME이 "neo4j"가 아니라 인스턴스 ID

**증상**
첫 Aura 호출 시 `ServiceUnavailable: Failed to DNS resolve address` (인스턴스
pause 상태) → resume 후 `DatabaseNotFound: database 'neo4j' does not exist` →
`NEO4J_DATABASE`를 인스턴스 ID로 교정 후 `AuthError: unauthorized`.

**원인**
세 가지가 겹쳤다.

1. Aura Free는 72시간 비활성 시 자동 pause. pause된 인스턴스는 DNS도 사라지므로
   에러가 wrong-password와 다른 시그널로 나옴.
2. Aura가 신규 프로비전한 인스턴스의 DB 이름은 기본 `neo4j`가 아니라 인스턴스 ID
   (URI 호스트의 첫 8자, 예: `d8af4c05`)와 같음. 드라이버는 `database` 파라미터를
   따로 받아야 한다.
3. **Aura의 username도 `neo4j`가 아니라 인스턴스 ID로 발급된다.** 인스턴스 생성
   시 다운로드되는 `Neo4j-XXX-Created-YYYY-MM-DD.txt` 파일이 SSOT —
   `NEO4J_USERNAME=<id>` / `NEO4J_PASSWORD=<auto-generated>` /
   `NEO4J_DATABASE=<id>` 세 줄을 그대로 쓰는 것이 안전.
   ("Aura는 username이 항상 neo4j" 라는 일반론은 신규 인스턴스에 대해 틀렸음 —
   콘솔 화면 또는 자격증명 파일을 SSOT로 삼을 것.)

**해결**
- `Settings`에 `neo4j_username` / `neo4j_database` 필드 추가, `.env.example`을
  Aura 자격증명 파일과 변수명 매칭(`NEO4J_USERNAME`, `NEO4J_DATABASE`).
- `.env`는 자격증명 파일 내용을 그대로 복사.
- 인스턴스 pause 가능성에 대비해 Day 4 첫 적재 단계에서 ping 명시적 호출.

**교훈**
- 외부 SaaS 자격증명은 콘솔 또는 다운로드 파일이 SSOT. 모델/문서가 추정한 일반
  관행보다 우선. 환경변수 디버깅 시 즉시 자격증명 파일 원문을 확인할 것.
- Aura Free 사용 중에는 pause-then-resume 흐름을 1차 진단 패턴으로 습관화
  (DNS 실패 = 일시 중지 가능성).

### 2026-05-03: DART 응답 percentage 필드에 주식 수 등 out-of-range 값 — 단일 회사가 전체 ingest 중단

**증상**
KOSPI 200 일괄 ingest 도중 `parse_dart_pct`가 `DartParseError: DART percentage
out of range [0, 100]: 10000000.0` (영풍 등)로 raise → 적재 스크립트 전체 abort.

**원인**
- 일부 공시는 `trmend_blce_qota_rt`(분기말 잔여 지분율) 필드에 지분율이 아니라
  주식 수 또는 다른 단위를 적어 둠. 데이터 품질 이슈.
- 우리의 엄격 검증(`parse_dart_pct`가 [0, 100] 범위 강제)은 의도된 동작 — 잘못된
  데이터를 조용히 그래프에 흘려보내지 않기 위함. 변경 대상 아님.
- 진짜 문제는 ingest 스크립트가 회사 단위 isolation 없이 `DartAPIError`만 catch
  하고 `DartParseError`는 흘려서 한 회사의 한 행이 전체를 깨뜨림.

**해결**
스크립트 `_fetch_all_candidates`에서 forward/reverse 호출 모두 `(DartAPIError,
DartParseError)`로 catch. 실패한 회사·방향만 warning 로그로 남기고 다음 회사로
계속. 현실 데이터 품질 잡음을 회사 단위로 격리.

**교훈**
- "loud not silent" 원칙(ADR/data-notes)에 따른 엄격 검증은 파서 레이어에서
  유지하되, **호출자(orchestration) 레이어는 회사·요청 단위로 isolation**.
- 새 외부 API에 대한 일괄 호출 스크립트는 처음부터 per-item try/except로 짤 것.
  단일 row가 전체 batch를 깨는 패턴은 KOSPI 200 같은 작은 universe에서도 한
  번은 반드시 만난다.

### 2026-05-03: KOSPI 200 1차 적재 결과 OWNS 0건 — 추출과 적재 사이 식별자 해소 단계 누락

**증상**
DART forward+reverse 200사 일괄 호출 후 후보 10,499건 추출 성공, 그러나 적재된
OWNS 엣지 0건. `OwnsLoadStats`가 모두 `dropped_endpoint_unresolved`로 분류
(0 outside_universe, 0 loaded). 그래프엔 회사 200개만 노드로 적재되고 관계는 비어 있음.

**원인**
DART 응답 구조 자체의 비대칭성과 적재 필터의 전제가 합치되지 않음.

- forward(`otrCprInvstmntSttus`): 호출 회사가 source(corp_code 있음), 투자 대상 회사가
  target(텍스트만, `inv_prm` 필드).
- reverse(`hyslrSttus`): 호출 회사가 target(corp_code 있음), 주주가 source(텍스트만, `nm` 필드).
- 즉 모든 후보는 한쪽 endpoint가 항상 텍스트.
- ADR 0007의 적재 필터는 "양 endpoint 모두 corp_code 있고 universe 안" 요구.
- 텍스트 → corp_code 해소 단계가 추출과 적재 사이에 없으면 모든 후보가 첫 조건에서
  탈락. architecture.md "v0 ER = 단순 키 매칭" 정의는 있었지만 코드로 구현되지 않은 상태.

**해결**
- `src/k_fingraph/resolve/owns.py` 신설 — `build_resolution_index` + `resolve_endpoints`.
  텍스트 endpoint의 정규화 이름을 corp_code 표에서 조회해 `corp_code` 필드를 채움.
- 모호한 이름은 universe 멤버 우선 → 상장(stock_code 있음) 보조 → 비상장(stock_code None)은
  의도적으로 인덱스에서 제외 (적재되면 안 되는 식별자가 universe 밖으로 빠지는 거짓
  분류를 피하기 위함, 진단기가 B-1로 분류하도록 양보).
- `scripts/load_v0.py` 흐름을 extract → **resolve** → upsert로 정정.
- 정책 결정을 ADR 0008로 영구화 (v0.5 진입 시 (A)/(B-1)/(B-2) 재측정 트리거,
  v2 ER 진입 시 supersede 대상).
- 재실행 결과: 후보 10,499 / 해소 587 / 적재 242 (그래프 236 — 같은 공시키 중복
  제거 후), 분류 A 338 / B-1 2,948 / B-2 6,964.

**교훈**
- 새 적재 파이프라인을 짤 때는 **end-to-end smoke를 적재 직후 결과 카운트로 검증**할
  것. 추출·필터 각각이 단위 테스트로 통과하더라도 사이를 잇는 단계의 누락은 결과
  0건으로만 드러남. 단위 테스트가 아닌 적재 결과 자체가 acceptance 기준이어야 함.
- 추출 출력 데이터 구조와 적재 필터가 가정하는 입력 구조가 일치하는지를 처음부터
  pipeline 다이어그램으로 그려보고 시작할 것. forward/reverse가 비대칭으로 corp_code를
  채워주는 사실은 data-notes에 기록돼 있었으나 적재 단계 설계 시점에 다시 떠올리지
  못한 것이 이 누락의 직접 원인.
- 살아있는 문서(architecture.md "v0 ER = 단순 키 매칭")가 정의만 박아두고 구현 책임이
  특정 sprint·코드 위치에 묶여 있지 않으면 누락된다. 정의가 들어간 시점에 어느 코드
  파일에서 누가 구현하는지를 같은 줄에 적어 두면 재발 위험 감소.

### 2026-05-03: 워크벤치 selectbox default — 의도 없는 정렬 첫 두 개 조합으로 잘못된 무결과 표시

**증상**
Streamlit 워크벤치 첫 진입 시 "공통 부모 찾기" 패널에서 종목 A·B selectbox
default가 sorted-by-name 첫 두 개(`삼성물산` / `삼성바이오로직스`)로 자동 선택되어
"두 종목을 동시에 직접 보유한 회사가 그래프에 없습니다." 메시지를 즉시 표시.
사용자가 패널에 들어가자마자 "데이터가 비어 있는가" 라고 오해할 수 있는 상태.

**원인**
- selectbox 기본 동작 — `index=0` 이면 첫 번째 옵션이 자동 선택됨.
- 정렬이 `name_kr ASC` 라 첫 두 개가 `삼성물산` / `삼성바이오로직스`로 들어왔는데,
  이 둘은 우연히 모-자 관계 (삼성물산이 삼성바이오로직스의 모회사)라 "공통 부모"
  쿼리 정의상 결과가 비어 있는 게 정상.
- 즉 쿼리 자체는 옳고, 데이터도 멀쩡한데, default 조합이 의도 없는 무의미한
  쿼리를 자동 실행한 결과를 사용자가 보게 됨.

**해결**
- `_ticker_selectbox`에 `default_index: int | None` 인자 + `placeholder` 인자
  추가. `default_index=None` 으로 호출하면 placeholder("종목을 선택하세요") 진입.
- 공통 부모 패널의 두 selectbox만 `default_index=None`으로 갈아 끼움. 자회사
  조회·2-hop 패널은 단일 입력이라 어떤 default든 의미 있는 결과를 내므로 그대로 유지.
- 두 종목이 모두 선택되기 전에는 쿼리를 실행하지 않고 "두 종목을 모두 선택하면
  결과가 표시됩니다" 캡션을 보여줌.

**교훈**
- UI default는 단순 "정렬 첫 항목"이 아니라 사용자 의도가 형성되기 전까지 쿼리가
  실행되지 않게 만드는 게 안전. 특히 다중 입력 워크플로우에서는 default 조합이
  의도되지 않은 의미를 가질 수 있다.
- 수동 브라우저 검증 시 — 어시스턴트가 미리 골라준 예시 입력으로만 확인하지 말고
  **사용자가 무지(無知)한 상태로 첫 진입할 때의 흐름**을 함께 확인해야 이런 케이스가
  잡힌다. 구현자(어시스턴트)가 default를 어떻게 잡았는지 알고 있는 경우 같은 함정에
  빠지기 쉬움.
- 단일 입력 패널은 default가 의미 있는 결과를 내므로 그대로 두는 게 사용자 경험에
  유리. 일률적 placeholder 적용 ≠ 항상 좋은 UX.

### 2026-05-04: v0.5 DoD "(A) 매칭 실패 → 0" 부분 달성 — 분류기 정의와 DART historical 보존 정책의 불일치

**증상**
v0.5 sprint DoD에 "(A) 매칭 실패 카운트 v0 baseline 338 → 0 또는 그에 가까운 값"
을 박았으나, universe를 KOSPI 200(200사) → 전체 KOSPI + KOSDAQ 보통주(2,659사)로
13배 확장한 후에도 (A)가 181건 잔존. 절반 가까이가 universe 확장으로 회복되지 않음.

**원인**
- `extract/owns_diagnostics.py`의 (A) 분류 정의가 `record.stock_code is not None`을
  "currently listed"의 proxy로 사용. 즉 "DART corp_code에 stock_code가 있으면 어떤
  거래소엔가 상장돼 있다"고 가정.
- 그러나 DART는 상장폐지·합병 등으로 더 이상 거래소에 존재하지 않는 회사도 historical
  stock_code를 보존. 잔여 (A) 181건의 sample(부산은행·경남은행·동양건설산업·경남기업·
  쌍용건설)이 모두 이 패턴 — 과거 상장 후 합병·인수·청산된 회사들.
- DoD를 박을 때 ADR 0007의 (A) 정의("universe만 넓히면 매칭됨")를 그대로 신뢰했고,
  분류 정의가 가정하는 도메인 모델("stock_code = currently listed")과 외부 시스템의
  실제 보존 정책("DART는 historical stock_code 보존") 사이의 불일치를 검증하지 않았음.

**해결**
- v0.5 안에서는 classifier 코드를 수정하지 않음 — 수정하면 v0 baseline과의 직접
  비교가 깨져 sprint의 측정 목적이 무너짐.
- 대신 ADR 0009 본문에 잔여 (A) 181건의 정체와 구조적 한계를 정직하게 기록 ("(A)
  잔여는 universe 확장으로 영구 해소 불가"). DoD 항목은 [~] 부분 달성 표기.
- classifier 정의 다듬기는 v2 ER sprint 영역(또는 별도 분류기 갱신 sprint)으로 이관.
  새 정의 후보: `stock_code is not None AND corp_code in current_listed_universe`.

**교훈**
- 외부 시스템의 식별자가 시간 차원의 의미("현재성")를 함의한다고 가정할 때는 그
  시스템의 보존 정책을 한 번 더 확인. DART처럼 historical 데이터를 보존하는 시스템은
  식별자의 "있음"이 "현재 활성"을 보장하지 않음.
- DoD에 외부 분류기·외부 표준의 결과 0% 잔여를 expectation으로 박기 전에, **잔여를
  만들 수 있는 구조적 원인이 분류 정의 자체에 있지 않은지** 한 번 점검. ADR 본문이
  "X 확장하면 다 잡힌다"고 단언해도, 그 확장이 분류 정의의 가정과 정합적인지를 별도
  검증.
- 부분 달성으로 끝난 결과는 ADR 본문에 "왜 100% 안 됐는지" 구조적 원인을 정직하게
  기록 → 다음 sprint가 정확한 baseline을 가짐. 숨기거나 DoD를 사후 완화하면 안 됨.
